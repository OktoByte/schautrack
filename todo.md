## Critical

- [ ] **[db/init.sql]** Missing index on `calorie_entries(user_id, entry_date)` — the most queried table and every dashboard/export query does a sequential scan. _Suggestion: add `CREATE INDEX IF NOT EXISTS calorie_entries_user_date_idx ON calorie_entries (user_id, entry_date);`_

- [ ] **[internal/handler/auth.go:181]** Registration `QueryRow.Scan(&exists)` error ignored. If the DB has a transient failure, `exists` stays `false` and registration proceeds — potentially creating a duplicate account despite the UNIQUE constraint (or worse, silently failing later). _Suggestion: check the error and return 500._

- [ ] **[internal/handler/auth.go:418-419]** Password reset ignores `Exec` errors on both the password hash update AND the mark-token-as-used. User could be locked out (hash not saved) or token remains reusable. _Suggestion: check both errors, return 500 on failure._

- [ ] **[internal/handler/settings.go:291]** Password change `Exec` error ignored. User thinks their password changed but it didn't persist. _Suggestion: check error and return 500._

- [ ] **[internal/handler/settings.go:358]** TOTP enable `Exec` error ignored. User thinks 2FA is active but the secret was never saved. _Suggestion: check error and return 500._

- [ ] **[internal/handler/ — systemic]** ~30 `Pool.Exec()` calls across handlers silently discard errors. Full list of affected files: `auth.go` (lines 260, 327, 329, 473, 474, 538, 702, 767, 768), `settings.go` (lines 55, 57, 111, 158, 230, 386, 531, 539), `entries.go` (lines 510, 536, 818, 821, 831, 834, 837, 861), `todos.go` (lines 32, 389, 419), `weight.go` (line 112), `admin.go` (line 123). _Suggestion: audit every Exec call; at minimum check error on writes that the user expects to succeed (password, email, 2FA, delete). Lower-stakes ones (sort order, SSE broadcast) can log-and-continue._

## High

- [ ] **[internal/handler/entries.go:818-822]** Import handler ignores `Exec` errors on DELETE statements inside the transaction. If delete fails, subsequent inserts create duplicates instead of replacing. Lines 831, 834, 837, 861 in the same function also ignore errors within the same tx. _Suggestion: check errors and rollback._

- [ ] **[internal/handler/auth.go:473-474]** Email verification: token marked as used and `email_verified` set to TRUE in two separate unchecked Exec calls. If first succeeds but second fails, token is consumed but email stays unverified — user can't retry. _Suggestion: wrap in a transaction or at minimum check errors._

- [ ] **[internal/handler/auth.go:767-768]** Email change: same two-unchecked-Exec pattern. Email update and token-mark-as-used can partially fail. _Suggestion: wrap in a transaction._

- [ ] **[internal/sse/broker.go:59]** `json.Marshal(payload)` error silently discarded — sends empty `data:` frames on marshal failure. _Suggestion: check error and skip the send._

- [ ] **[internal/middleware/ratelimit.go:63-76]** Rate limiter `entries` map grows unbounded within a window. Behind a CDN or with spoofed X-Forwarded-For, memory grows until the next cleanup tick. _Suggestion: add a max entries cap and early-reject when full._

- [ ] **[db/init.sql:105]** `todo_completions.user_id` has no FOREIGN KEY to `users(id)`. If a user is deleted, their completions are orphaned (the todo FK cascades, but direct user deletion doesn't clean up completions for todos owned by others). _Suggestion: add `REFERENCES users(id) ON DELETE CASCADE`._

- [ ] **[client/src/pages/Dashboard/EntryForm.tsx:38-40]** Uses `parseFloat()` for macro inputs instead of the project's own `parseAmount()`. `parseFloat("12abc")` silently returns 12. _Suggestion: use `parseAmount()` from `@/lib/mathParser` for consistent validation._

- [ ] **[client/src/pages/Dashboard/EntryForm.tsx:54]** `Number(amount)` can produce `NaN` for empty/non-numeric input, sent to server without validation. _Suggestion: validate before submission; show user error if invalid._
